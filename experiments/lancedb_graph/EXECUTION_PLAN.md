# LanceDB 图存储实验执行计划

## 1. 实验目标

本轮实验聚焦于一个明确目标：

在 `LanceDB` 中采用“节点表 + 边表”的图存储模型，通过索引设计、物理布局优化和查询执行优化，提高图检索效率，并与其他图存储方案进行统一对比。

需要特别说明的是：

- `三元组表` 不再作为本轮实验的候选方案或对比方案
- 本轮在 `LanceDB` 中的基础方案就是 `节点表 + 边表`
- 后续所有优化实验，都建立在该基础模型之上


## 2. 实验范围

本轮实验重点回答以下问题：

1. 在 `LanceDB` 中采用 `nodes + edges` 的图模型后，单跳和多跳查询的性能上限如何
2. 哪些索引设计或数据布局策略，能够显著提升图检索效率
3. 在查询性能、写入成本、存储占用和更新代价之间，最合适的折中点是什么
4. 与其他图存储方案相比，`LanceDB nodes + edges` 的优势和短板分别是什么

本轮实验不以“做一个完整图库接口”作为第一目标，而是优先保证：

- 建模清晰
- 实验可复现
- 指标可解释
- 优化路径可拆分


建议各目录职责如下：

- `utils/`: 公共工具函数


## 4. 基础数据模型

本轮 `LanceDB` 的基础模型固定为两张核心表。

### 4.1 节点表 `nodes`

建议至少包含以下字段：

- `node_id`: 节点唯一标识
- `node_type`: 节点类型
- `degree_out`: 出度
- `degree_in`: 入度
- `attrs_json`: 节点属性

如后续需要扩展语义检索或混合检索，可预留：

- `vector`: 节点向量

### 4.2 边表 `edges`

建议至少包含以下字段：

- `edge_id`: 边唯一标识
- `src_id`: 起点节点 ID
- `dst_id`: 终点节点 ID
- `edge_type`: 边类型
- `src_type`: 起点节点类型
- `dst_type`: 终点节点类型
- `attrs_json`: 边属性
- `timestamp`: 可选，用于时间过滤或排序

### 4.3 基础要求

基础实现阶段不追求复杂优化，先保证：

- 数据正确导入
- 节点与边能正确关联
- 单跳和多跳查询语义正确
- benchmark 可以统一调用


## 5. 实验分阶段计划

### 阶段一：建立实验骨架与基础实现

目标：

- 建立新实验目录
- 固定配置和文件组织方式
- 完成 `nodes` / `edges` 数据转换
- 实现基础版 `LanceDB nodes + edges`

交付物：

- `README.md`
- `config.py`
- 数据转换脚本
- 基础存储实现
- 基础查询接口

阶段一建议完成的接口：

- `build()`
- `load()`
- `query_out_neighbors(node_id)`
- `query_in_neighbors(node_id)`
- `query_neighbors(node_id)`
- `query_k_hop(node_id, k)`
- `stats()`

说明：

- 阶段一只做“最朴素但正确”的实现
- 不引入复杂索引
- 不做多种布局
- 先建立所有优化实验的共同基线


### 阶段二：单跳查询优化

目标：

- 围绕单跳邻居查询设计边表布局策略
- 评估不同物理组织方式的收益

优先实验项：

1. `edges` 按 `src_id` 排序或聚簇
2. `edges` 按 `dst_id` 排序或聚簇
3. `edges` 按 `(src_id, edge_type)` 排序
4. `edges` 按 `(dst_id, edge_type)` 排序
5. 双表策略：正向边表 + 反向边表

核心对比问题：

- 出边查询是否明显受益于 `src_id` 聚簇
- 入边查询是否需要单独优化
- 类型过滤是否值得放入主排序键
- 双表设计带来的写放大是否可接受

交付物：

- 单跳查询 benchmark
- 不同布局方案的结果表
- 初步结论：哪种布局最适合作为后续默认方案


### 阶段三：邻接索引与局部缓存优化

目标：

- 在 `nodes + edges` 基础上引入额外辅助结构
- 提高高频单跳与多跳扩展效率

优先实验项：

1. 邻接摘要表
2. 正向邻接索引
3. 反向邻接索引
4. 按边类型组织的邻接摘要
5. 常用节点或热点节点邻接缓存

邻接索引表可考虑包含：

- `node_id`
- `out_degree`
- `in_degree`
- `out_edge_ids` 或 `out_edge_ranges`
- `in_edge_ids` 或 `in_edge_ranges`

核心对比问题：

- 邻接摘要表对单跳查询的收益有多大
- 邻接索引对多跳扩展是否有效
- 对高度节点是否会造成索引行过大
- 写入和更新代价是否明显上升

交付物：

- 邻接索引实现
- 邻接索引与基础边表的对比结果
- 高频节点和高度节点的性能分析


### 阶段四：多跳查询执行优化

目标：

- 优化 `2-hop` 和 `3-hop` 查询路径
- 降低 frontier 扩展和重复扫描带来的成本

优先实验项：

1. 朴素 BFS
2. 批量 frontier 查询
3. 分层去重
4. 邻接摘要 + 回表模式
5. 高度节点特殊处理

核心关注指标：

- 每跳扫描边数
- 每跳中间结果规模
- 去重成本
- 总耗时

交付物：

- 多跳查询 benchmark
- 单跳优化与多跳优化组合结果
- 多跳执行策略对比分析


### 阶段五：外部图存储方案对比

目标：

- 将 `LanceDB nodes + edges` 与其他图存储方案进行统一对比

建议至少纳入以下系统：

1. `Neo4j`
2. 一个轻量图数据库或图查询系统，例如 `Kuzu`
3. 可选：列式关系方案，例如 `DuckDB + Parquet`

说明：

- 本轮外部对比不再纳入旧的 `LanceDB 三元组表`
- 对比重点是“图存储方案之间的差异”
- `LanceDB` 的对比对象应覆盖至少一种原生图数据库

交付物：

- 对比系统适配层
- 统一 benchmark 运行脚本
- 横向对比结果与总结


## 6. Benchmark 设计

### 6.1 查询任务

建议统一测试以下查询任务：

1. 节点查找：按 `node_id` 获取节点
2. 单跳出邻居查询
3. 单跳入邻居查询
4. 单跳双向邻居查询
5. 按 `edge_type` 过滤的单跳查询
6. 按 `node_type` 过滤的单跳查询
7. `2-hop` 邻居扩展
8. `3-hop` 邻居扩展
9. 子图抽取：给定起点和跳数
10. 可选：属性过滤 + 图扩展混合查询

### 6.2 数据规模

建议分层测试，而不是只跑全量：

- 小规模
- 中规模
- 大规模

如果沿用当前数据集，可按边数做分级，例如：

- `1e4`
- `1e5`
- `1e6`
- 全量

### 6.3 节点分层

建议继续沿用按节点关联度分层的思路：

- 低度节点
- 中度节点
- 高度节点

需要保证所有方案使用相同测试样本集。

### 6.4 指标

建议统一记录以下指标：

- 平均延迟
- `P50`
- `P95`
- `P99`
- 吞吐量 `QPS`
- 写入耗时
- 索引构建耗时
- 存储占用
- 更新代价
- 中间结果规模
- 可选：扫描批次数或扫描行数

### 6.5 运行规范

为了保证结果可解释，建议固定以下规则：

- 冷启动与热启动分开测试
- 固定随机种子
- 每类查询重复多次
- 使用同一组测试节点
- 使用同一套过滤条件
- 在统一机器环境中执行


## 7. 建议优先实现的优化项

为了避免实验面铺得过大，建议优先级如下：

### 第一优先级

- `src_id` 聚簇边表
- `dst_id` 聚簇边表
- 正向边表 + 反向边表

原因：

- 实现成本较低
- 对单跳性能影响通常最直接
- 能快速建立“布局是否有效”的第一轮结论

### 第二优先级

- 邻接摘要表
- 邻接索引表
- frontier 批量查询

原因：

- 对多跳查询和热点节点收益潜力较高
- 更能体现 `LanceDB` 存图方案的差异化设计

### 第三优先级

- 高度节点专项优化
- 热点节点缓存
- 边类型分区
- 节点类型分区

原因：

- 这类优化更依赖实际查询分布
- 适合在前两轮结果出来后再决定是否继续深入


## 8. 统一接口建议

建议不同存储实现统一暴露相同接口，降低 benchmark 层复杂度。

建议接口如下：

```python
class GraphStore:
    def build(self): ...
    def load(self): ...
    def stats(self): ...
    def query_out_neighbors(self, node_id, edge_type=None): ...
    def query_in_neighbors(self, node_id, edge_type=None): ...
    def query_neighbors(self, node_id, edge_type=None): ...
    def query_k_hop(self, node_id, k, edge_type=None): ...
```

建议对应模块示例：

- `storage_models/lancedb_graph_basic.py`
- `storage_models/lancedb_graph_dual_edges.py`
- `storage_models/lancedb_graph_adjacency.py`
- `storage_models/neo4j_graph.py`
- `storage_models/kuzu_graph.py`

这样可以让 benchmark 代码更多关注“测什么”，而不是“每个系统怎么调用”。

## 11. 关键风险

在执行过程中需要重点关注以下风险：

- 如果查询实现仍然大量退化为全表加载再过滤，会掩盖真实存储设计效果
- 如果布局优化、执行优化和缓存策略同时叠加，实验结果会很难解释
- 高度节点的邻接索引可能非常大，需要单独策略
- 双表和邻接索引会提升查询性能，但也会明显增加写放大和更新复杂度
- 外部对比系统如果查询语义不完全一致，会影响结论可信度
